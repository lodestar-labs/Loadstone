Loadstone

Import data of any shape — XML, JSON, CSV — into a relational database, declaratively.

Architecture & capabilities overview · github.com/KadjiProjects/Loadstone · .NET 10 · MIT

1. The core idea

Every integration team eventually hand-writes the same program: parse a file, walk its hierarchy, resolve reference codes, insert parent rows, wire up foreign keys, insert children, and cope with the bad rows. Loadstone replaces that entire class of one-off importers with a single declarative JSON manifest per dataset. The manifest is the only thing you author. Everything else is derived from it:

You declare (once)Loadstone derives
Entity hierarchy, field names and types, natural keys, lookup rules, queue settings Format parsing expectations · validation · type conversion · staging DDL · merge/upsert SQL · foreign-key wiring · target-table CREATE scripts · a dedicated durable queue · rejection reporting

2. System architecture

Five projects, layered so that every external concern — file format, database engine, queue transport, file storage, reference-data source — sits behind a small interface defined in a dependency-free core.

3. The import pipeline

Each job is one streaming pass. Records flow from the reader through the ordered step chain; a record tree that picks up an error anywhere is diverted to the rejection store, and everything else streams to the writer. Memory is bounded by a single record subtree, never the file, so multi-gigabyte inputs are fine.

4. The write path: hierarchy without ORM round trips

The writer never touches rows one at a time. Batches of record trees are flattened per entity, bulk-copied into temp staging tables, and merged into the targets level by level — parents first. Each merge's OUTPUT clause maps every staged row to the database key it received (or already had), and those keys become the children's foreign keys in the next level. One transaction per batch; natural keys make re-imports idempotent.

5. Capabilities

6. Bring your own lookup database — the "company X" case

Reference-data resolution is where importers usually stop being generic: real organizations already have a lookup database, and it doesn't look like anyone else's. Loadstone treats that as the normal case, with three tiers of integration — from zero code to full control.

The scenario (taken from the project Loadstone generalizes): a fisheries research agency imports scientific sampling data. Its reference database — call it RES — holds a generic pair of tables, tblCodeType (about ninety named lists: SpecWoRMS species codes, ISO_3166 countries, gear types, harbours…) and tblCode (the codes themselves). Incoming files carry strings like "126417" or "DK"; the target tables store the matching tblCodeID integer. In the legacy importer this mapping was a hard-coded C# enum that had to be kept in sync with the database by hand, and an unknown code crashed the entire file.

Tier 1 — configuration only: point Loadstone at the existing tables

Company X's lookup database becomes a provider by adding a query to appsettings.json (or an environment variable / Key Vault entry). No build, no deployment of new code: 

"Loadstone": {
  "SqlLookups": [
    {
      "key": "res-codes",
      "connectionString": "Server=res-db;Database=RES;Integrated Security=True;TrustServerCertificate=True",
      "query": "SELECT c.tblCodeID
                FROM dbo.tblCode c
                JOIN dbo.tblCodeType t ON t.tblCodeTypeID = c.tblCodeTypeID
                WHERE t.CodeType = @List AND c.Code = @Value AND c.Deprecated = 0"
    }
  ]
}

The contract is small: the query receives @List (the manifest's list name) and @Value (the raw source string) and returns one scalar on a hit, no rows on a miss. It can live in a different database, a different server, even a different vendor's driver later — the pipeline neither knows nor cares.

Manifest fields then select the provider and list by name, and choose a policy per field:

{ "name": "SAspeciesCode",
  "lookup": { "provider": "res-codes", "list": "SpecWoRMS", "onMissing": "rejectRecord" } },

{ "name": "VDcountry",
  "lookup": { "provider": "res-codes", "list": "ISO_3166", "onMissing": "rejectFile" } }

That last line reproduces the legacy system's strictness where it's wanted (a bad country code means the file itself is suspect) while the species field above only quarantines the affected sample — with the source line, the raw code, and the reason recorded for the data manager. The ninety-entry enum is gone; the lists are whatever tblCodeType says they are, today.

Tier 2 — one small class: lookups that aren't a SQL query

If company X later moves species validation behind a web service, the provider becomes an ordinary class instead of a query — same manifest, same policies, one changed registration: 

public sealed class WormsApiLookup(HttpClient http) : ILookupProvider
{
    public string Key => "worms-api";

    public async ValueTask<LookupResult> ResolveAsync(
        string list, string value, bool caseInsensitive, CancellationToken ct = default)
    {
        var response = await http.GetAsync($"/rest/AphiaRecordByID/{value}", ct);
        return response.IsSuccessStatusCode
            ? LookupResult.Hit(int.Parse(value))
            : LookupResult.Miss;
    }
}

// Program.cs
builder.Services.AddLoadstone()
    .UseSqlServer()
    .AddLookupProvider<WormsApiLookup>();

Providers can optionally implement CreateAsync to support the autoCreate policy (the built-in code-list provider does); providers that don't simply reject it. Caching, per-field policy handling, and miss bookkeeping stay in the resolver — a provider only ever answers "what does this value mean?".

How a lookup field resolves

Why this matters: in the legacy importer this whole area was a hard-coded enum, a case-sensitive dictionary, and a crash on the first unknown code. In Loadstone it is a per-field, per-dataset decision expressed in data — and pointing a new company's lookup database at the system is an appsettings entry, not a fork.

7. Why this could be popular on GitHub

Honesty first: nothing guarantees stars — six-figure counts put a project in the company of the largest tools in the industry and take years, visibility, and luck. What follows is why Loadstone has the structural ingredients that widely-starred infrastructure projects tend to share.
  1. A universally felt pain. "Get this file into my database, properly" is one of the most common tasks in backend, data, and integration work. The audience is not a niche. And the specific corner Loadstone owns — nested XML/JSON/CSV into multi-table hierarchies with row-level forensics — is underserved: most ETL platforms treat it as an afterthought behind heavyweight infrastructure.
  2. Time-to-wow under five minutes. docker compose up, open Swagger, curl a sample file, watch rows and a job timeline appear — database and schema created automatically. Projects win stars in the first five minutes of a visitor's attention or not at all.
  3. A shareable artifact at the center. Manifests are small JSON documents — easy to gist, blog about, review in pull requests, and collect into a community library ("here's a manifest for format X"). Config-as-contract is the same dynamic that helped docker-compose and GitHub Actions spread.
  4. Credible engineering underneath. Streaming everywhere, set-based bulk writes, no per-row reflection, a broker-less durable queue, first-class OpenTelemetry. Experienced developers star what they would defend in an architecture review — and they are the ones whose stars bring more stars.
  5. An obvious contribution surface. The provider seams (PostgreSQL, MySQL, Parquet, Blob Storage, Service Bus) are shaped exactly like good first contributions, and the roadmap invites them. Projects grow when outsiders can add real value without touching the core.
  6. Low adoption risk. MIT license, one container, no dependencies beyond the database you already run, and an escape hatch at every layer. Easy to try, easy to leave — which, paradoxically, is what makes teams stay.

What it still needs to climb

A hosted demo or an animated GIF at the top of the README; two or three real-world manifest showcases; benchmark numbers ("N GB of XML in M minutes"); the PostgreSQL provider, which roughly doubles the addressable audience; and visible, responsive maintenance once public — how the first twenty issues are handled is what converts visitors into advocates.

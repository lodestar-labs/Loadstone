1. The model in one minute

Imports are asynchronous. You never wait on an upload — you get a job id and ask about it.

A dataset (declared once, in a JSON manifest) describes the shape of the data and its target tables. Uploading a file to a dataset creates an import job on a persistent queue; a worker picks it up, streams the file through validation → lookups → the database writer, and records the outcome plus a per-row rejection log. Your integration therefore has exactly two moving parts: submit (one POST) and observe (poll the job until it reaches a terminal status).

2. Authentication

Out of the box the API is open (suitable for a private network). Setting one configuration value locks every /api route behind a shared key:

# environment variable form
Loadstone__ApiKey=your-long-random-secret

From then on, every request must carry the header — a missing or wrong key gets 401 { "error": "Missing or invalid X-Api-Key header." }:

curl -H "X-Api-Key: your-long-random-secret" http://localhost:8080/api/datasets

The comparison is constant-time server-side. The dashboard prompts for the key once and remembers it in the browser. Health probes (/health/live, /health/ready) stay open by design — orchestrators need them before they could know any secret.

3. Uploading a file

curl -H "X-Api-Key: $KEY" \
     -F "file=@orders-2026-07.xml" \
     "http://localhost:8080/api/datasets/orders/imports?requestedBy=nightly-feed"

The response is 202 Accepted — the file is stored and queued, nothing has been imported yet:

{
  "jobId": "0e1f9a3c-…",
  "correlationId": "…",
  "queue": "orders",
  "status": "Pending"
}

Details worth knowing:

  • Format is inferred from the file extension (.xml, .json, .csv, .zip); pass ?format=… explicitly when the extension is missing or misleading. A dataset may restrict which formats it accepts — violations are a 400 naming the allowed list.
  • requestedBy is a free-text caller tag stored on the job — invaluable when several systems feed the same dataset.
  • The Location header points at the job resource: /api/imports/{jobId}.
  • A 404 means the dataset isn't registered; a 400 names the exact problem (unknown format, no reader, format not accepted).

4. The job lifecycle — what the statuses mean

StatusTerminal?MeaningYour reaction
PendingnoQueued, not yet picked up.Keep polling.
ProcessingnoA worker is streaming the file.Keep polling.
SucceededyesEvery record imported.Done.
CompletedWithRejectionsyesAccepted records were written; some rows were rejected (validation, lookups). This is a success status for the file and a to-do list for the data.Fetch /rejections, route them to whoever owns the source data.
FailednoAn attempt failed (infrastructure, database). It will be retried automatically until attempts are exhausted.Keep polling — do not resubmit; that would double-import on recovery.
DeadLetteredyesAll attempts exhausted; requires an operator.Alert. Inspect /events for the failure trail; resubmit only after the cause is fixed.
The distinction that matters most: CompletedWithRejections and Failed are opposites, not neighbors. The first means "the import finished; these specific rows need fixing." The second means "nothing is final yet; the system is still trying." Automation that treats them alike either re-sends files that already landed or ignores rows that silently went missing.

5. Following a job to completion

# poll until terminal
GET /api/imports/{jobId}          → { id, dataset, status, attempt, … }

# when CompletedWithRejections: what was rejected, and why
GET /api/imports/{jobId}/rejections?top=1000
→ row-by-row: the record path, field, message, and the raw offending value

# the full audit trail (attempts, errors, timings)
GET /api/imports/{jobId}/events

A robust polling loop in any language:

// C# sketch — the shape matters more than the language
while (true)
{
    var job = await http.GetFromJsonAsync<JobView>($"/api/imports/{jobId}");
    switch (job.Status)
    {
        case "Succeeded":                 return Outcome.Clean;
        case "CompletedWithRejections":   return await CollectRejections(jobId);
        case "DeadLettered":              return Outcome.NeedsOperator;
        // Pending, Processing, Failed → still in flight (Failed retries automatically)
    }
    await Task.Delay(TimeSpan.FromSeconds(5));   // 2–10s is a sensible cadence
}

Poll every few seconds for interactive flows, every 30–60 s for nightly feeds. There are no completion webhooks yet (they're on the roadmap) — polling is the supported mechanism, and the job store is indexed for it.

Everything below requires the X-Api-Key header when a key is configured. Interactive documentation for the same surface: Swagger UI at /swagger.

7. Integration patterns

7.1 A service submits and awaits

The bread-and-butter integration: your backend receives or produces a file, POSTs it, persists the returned jobId next to its own record, and a background poller drives the state machine from section 5. Persisting the jobId is the important habit — it makes the import resumable across your own restarts and idempotent to operator questions ("did file X land?").

7.2 Scheduled batch feeds

A nightly job uploads whatever appeared in a drop folder, then polls each job to terminal state and writes a summary (imported / rejected / dead-lettered) to its own log or alerting channel. Everything is one curl + one polling loop; requestedBy tags the run for traceability.

7.3 Validate first, then import (with Rowvane Gate)

When the sender should get feedback before anything touches the database, put Rowvane Gate in front: validate the file against a ruleset, only forward it to Loadstone when the report is clean. The complete recipe — including running both services side by side — is the validate & import guide.

7.4 Receiving exports from Fathom

Hierarchical CSV produced by Fathom (a zip of {Entity}.csv files with _key/_parentKey columns) is exactly the shape Loadstone's hierarchical CSV import reads — database-to-database transfer is a Fathom export POSTed straight to a Loadstone dataset.

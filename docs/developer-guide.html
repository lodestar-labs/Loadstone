Loadstone Developer Guide

How the system is built, why it is built that way, and where to change it.

This guide is for engineers who will maintain and extend Loadstone. It assumes you can read C# but know nothing about this codebase. It explains the architecture first, then walks each major component, quoting the load-bearing parts of the code and — more importantly — the reasoning behind them. Read sections 1–3 in order; the rest can be read on demand.

1. What Loadstone is

Loadstone imports hierarchical, tabular data of any shape — nested XML, JSON, or curated CSV — into a relational database, driven by a single declarative document per dataset called a manifest. The manifest describes the entity hierarchy, field types, keys, and reference-data rules; everything else (parsing, validation, staging DDL, merge SQL, foreign-key wiring, queueing, error reporting) is derived from it. There is no per-dataset code.

The mental model that explains almost every design decision:

The core contract A file arrives → it becomes a durable job on a queue → a worker streams the file through a pipeline of steps → clean records are written to the database, bad records are diverted to a rejection store with full source context. Jobs are at-least-once: the same file may be processed more than once (retry, reclaim), so every write path must be safe to repeat.

Two properties are load-bearing throughout and worth internalizing now:

  • Streaming. Records flow one subtree at a time from parser to writer. Memory is bounded by a single root record's subtree, never the file. A 500 MB file imports in roughly constant memory.
  • Fault isolation. One malformed record never aborts a good file. Bad records are quarantined; the rest import. This is the single biggest behavioral difference from the hand-written importers Loadstone replaces.

4. The manifest model

Loadstone.Core/Manifests/*.cs

The manifest is the heart of the system: a tree of EntityDefinition, each holding FieldDefinitions, rooted in a DatasetManifest. It is plain C# with JSON (de)serialization, and it is the single source of truth every downstream component reads.

DatasetManifest

Top-level: a name, queue settings, source settings, and the Root entity. Two methods matter more than the rest.

EnumerateEntities() walks the tree breadth-first, parents before children. This ordering is not cosmetic — the writer relies on it to merge parents before the children that reference them:

public IEnumerable<EntityDefinition> EnumerateEntities()
{
    var queue = new Queue<EntityDefinition>();
    queue.Enqueue(Root);
    while (queue.Count > 0)
    {
        var entity = queue.Dequeue();
        yield return entity;                     // parent emitted…
        foreach (var child in entity.Children)
            queue.Enqueue(child);                // …before its children
    }
}

Validate() returns a list of every problem found (never throws, never stops at the first). This is deliberate: an author fixing a manifest should see all errors at once. It checks structural rules (unique entity names, children have a parentKeyColumn, natural-key columns exist) and — crucially — rules that would otherwise produce broken SQL only at runtime:

  • a field column colliding with keyColumn or parentKeyColumn (those are database-generated / auto-wired);
  • decimal precision/scale out of SQL Server's range;
  • maxLength outside 1–4000;
  • a string natural-key column longer than 850 chars (exceeds the index key-size limit).
Why validation is so defensive Every rule in Validate() exists because a missing check once turned into a cryptic CREATE TABLE or MERGE failure deep in the writer. The principle: catch the author's mistake at registration time, with a clear message, not at import time against the database.

EntityDefinition

One level of the hierarchy = one relational table.

PropertyMeaning
NameWhat readers match: XML element, JSON property, or CSV file name (case-insensitive).
Table, SchemaTarget table. QualifiedTable returns the bracket-escaped [schema].[table].
KeyColumnSurrogate identity PK (database-generated; never supplied in data).
ParentKeyColumnFK to the parent's key. Null on the root; required on children.
NaturalKeyColumns that identify a row for upsert, scoped within the parent. Empty ⇒ always insert.
ChildrenNested entities, any depth.

FieldDefinition

A source value and where it lands. Key fields: Name (source), Column (target, defaults to Name), Type (a FieldKind enum: String, Int32, Int64, Decimal, Double, Boolean, Date, DateTime, Time, Guid), Source (Element / Attribute / Constant), plus Required, MaxLength, Precision/Scale, Format, Default, and an optional Lookup.

LookupSettings

Turns a raw code into a resolved value (see §10). Carries the list name, the provider key ("codelist" by default), the OnMissing policy (RejectRecord, RejectFile, UseDefault, AutoCreate), and a ValueKind (the resolved value's type, default Int32 — this drives the target column type for lookup fields).

Serialization & the IDataset interface

ManifestSerializer centralizes the JsonSerializerOptions (camelCase, case-insensitive, string enums, comments allowed) and validates on deserialize — you cannot construct an invalid manifest from JSON. Datasets can also be declared in code via IDataset, which uses a C 11 static abstract member so the manifest is checked at compile time:

public interface IDataset
{
    static abstract DatasetManifest Manifest { get; }
}

5. The record model

Loadstone.Core/Records/DataRecord.cs

DataRecord is the canonical in-flight shape every reader produces and the whole pipeline consumes. Format-specific detail dies at the reader boundary; nothing after it knows whether the source was XML, JSON, or CSV.

public sealed class DataRecord
{
    public required EntityDefinition Entity { get; init; }
    public SourceLocation Location { get; init; }         // line / path, for rejections
    public Dictionary<string, string?> Raw  { get; }      // by field name — kept for reports
    public Dictionary<string, object?> Values { get; }     // by target column — converted
    public List<DataRecord> Children { get; }
    public List<RecordIssue> Issues { get; }
    public object? DatabaseKey { get; set; }               // filled by the writer after merge

    public bool HasErrors     => Issues.Any(i => i.Severity == IssueSeverity.Error);
    public bool TreeHasErrors => SelfAndDescendants().Any(r => r.HasErrors);
}

Three things to understand here:

  • Two dictionaries. Raw holds original source strings keyed by field name; Values holds converted CLR values keyed by column name. Readers fill Raw; the validate step fills Values. Rejection reports read Raw so operators see exactly what was in the file.
  • TreeHasErrors vs HasErrors. The engine rejects on the tree: if any descendant is bad, the whole root subtree is diverted. This keeps a parent and its children atomic — you never write a parent whose child failed.
  • DatabaseKey. Mutable, set by the writer after a row is merged. This is how a parent hands its generated identity down to its children as their foreign key.

6. Value conversion

Loadstone.Core/Records/ValueConverter.cs

Converts raw source strings to CLR values. The performance-critical detail: a converter delegate is compiled once per FieldDefinition and cached, so the per-row hot path is a single delegate call — no reflection, no per-row type switch.

private static readonly ConcurrentDictionary<FieldDefinition, Func<string, object?>> Cache = new();

public static bool TryConvert(FieldDefinition field, string? raw, out object? value, out string? error)
{
    var effective = string.IsNullOrWhiteSpace(raw) ? field.Default : raw;
    if (string.IsNullOrWhiteSpace(effective)) { value = null; return true; }   // null/default
    try
    {
        value = Cache.GetOrAdd(field, BuildConverter)(effective.Trim());
        return true;
    }
    catch (Exception ex) when (ex is FormatException or OverflowException or ArgumentException)
    {
        value = null;
        error = $"Value '{Truncate(effective)}' is not a valid {field.Type}.";
        return false;                                                          // never throws to caller
    }
}

Two behaviors worth remembering: parsing is always invariant-culture (so 1.5 means one-and-a-half regardless of the server's locale), and TryConvert never throws — a bad value returns false with a message, which becomes a record rejection rather than an exception. Decimals use NumberStyles.Float so JSON exponent notation (1.5e3) parses.

7. Source readers

Loadstone.Readers/*

All three implement one interface, and the contract's key clause is stream:

public interface ISourceReader
{
    string Format { get; }                       // "xml" | "json" | "csv"
    IReadOnlyList<string> Extensions { get; }     // for format inference from file name
    IAsyncEnumerable<DataRecord> ReadAsync(Stream stream, DatasetManifest manifest, CancellationToken ct);
}

XmlSourceReader — a streaming state machine

Uses a forward-only XmlReader and materializes one root-entity subtree at a time. It matches the root entity's element name anywhere in the document (or, if source.xml.rootElement is set, only inside that wrapper element). Elements matching a child entity recurse; elements matching a field become raw values; everything else is skipped — so a partner adding extra elements never breaks your import. Line numbers are captured via IXmlLineInfo for rejection reports.

Editing the XML reader safely It is a hand-written state machine over a char/element stream. The invariant that keeps it correct: every branch of the inner loop must advance the reader, and after reading an entity subtree the outer loop must not skip the node the sub-read stopped on (that's the readNext = false dance). A malformed field element (nested markup where text was expected) is caught and turned into a SourceFormatException, not left to bubble.

JsonSourceReader

A top-level array is streamed element-by-element via JsonSerializer.DeserializeAsyncEnumerable — bounded memory at any size. A top-level object that wraps the array (via source.json.rootProperty or a property named after the root entity) is also supported, but that path parses the whole document.

Known trade-off The wrapped-object path is not streaming. It is documented in the class summary and the manifest reference. Very large exports should use a top-level array. If you ever need streaming for wrapped objects, that's a Utf8JsonReader rewrite — non-trivial.

CsvSourceReader

Two shapes. A dataset whose root has no children imports from a single CSV file. A hierarchical dataset imports from a zip archive with one EntityName.csv per entity, where each row has a _key and child rows carry a _parentKey that links them to their parent. The linking happens in memory after reading; duplicate or missing keys reject the affected record with a diagnostic. The underlying CsvParser is a dependency-free RFC 4180 state machine (quoted fields, escaped quotes, embedded newlines) that survives 8 KB buffer refills — all its state lives outside the read loop, which is the property to preserve if you touch it.

8. The import engine

Loadstone.Runtime/ImportEngine.cs

ImportEngine.RunAsync orchestrates one job as a single streaming pass. This is the class to read first if you want to understand how the pieces connect. Its shape:

// 1. Resolve the reader for the job's format and the ordered steps for this manifest.
var reader = readers.First(r => r.Format == job.Format);
var orderedSteps = steps.Where(s => s.AppliesTo(manifest)).OrderBy(s => s.Order).ToArray();

// 2. Open the file and hand a *lazy* stream of accepted roots to the writer.
await using var stream = await fileStore.OpenReadAsync(job.FileReference, ct);
var accepted = ProcessAsync(reader, stream, context, orderedSteps, pendingRejections, ct);
writeResult = await writer.WriteAsync(context, accepted, ct);   // writer pulls records through

The engine does not build a list and pass it on. ProcessAsync is an async iterator; the writer pulls records through it. Each root that comes out of the reader is run through every step, then:

foreach (var step in orderedSteps)
    await step.ExecuteAsync(context, root, ct);

if (root.TreeHasErrors)
{
    context.AddRecordsRejected(treeSize);
    CollectRejections(job.Id, root, pendingRejections);   // → rejection store
    continue;                                             // NOT yielded to the writer
}
yield return root;                                        // clean → flows to the writer

So "validate → resolve → split → write" is not four passes over the data; it is one pass where each record is fully processed and then either yielded or diverted. Rejections are flushed to the store in batches. On a retry (attempt > 1) the engine first clears the previous attempt's rejections so reports reflect the latest run.

9. Pipeline steps

Loadstone.Runtime/Steps/*

A step is a pluggable stage applied to every root record between parsing and writing.

public interface IImportStep
{
    string Name { get; }
    int Order => 500;                                       // built-ins: Validate=100, Lookups=200
    bool AppliesTo(DatasetManifest manifest) => true;
    ValueTask ExecuteAsync(ImportContext context, DataRecord root, CancellationToken ct);
}

Steps run in ascending Order. They mark problems via record.AddError(...); a record whose subtree accumulates an error is diverted to the rejection store instead of the database. Two are built in:

ValidateStep (Order 100)

Converts raw strings to typed values (filling record.Values), applies defaults and constants, and enforces required fields, max lengths, and required child entities. It never throws — every failure is an AddError. Lookup fields are skipped here except for a presence check; their conversion is the lookup step's job.

LookupStep (Order 200)

Resolves every lookup field through the ILookupResolver and applies the outcome: write the resolved value, reject the record, or (for RejectFile) throw LookupRejectedFileException to abort the whole import. AppliesTo returns false when the manifest has no lookups, so datasets without reference data skip it entirely.

Adding your own step Implement IImportStep, pick an Order (＞200 runs after the built-ins, before the writer), and register it: builder.AddStep<MyStep>(). Use it to enrich values, apply cross-field rules, or veto records. See §18.

10. Lookups

Loadstone.Core/Lookups/* · Loadstone.Runtime/Lookups/* · Loadstone.SqlServer/CodeList*, SqlQueryLookupProvider

Lookups turn a raw code ("DK") into a stored value (an id). The design separates what a value means (a provider) from policy and caching (the resolver).

ILookupProvider — the pluggable part

public interface ILookupProvider
{
    string Key { get; }                                     // manifests reference this
    ValueTask<LookupResult> ResolveAsync(string list, string value, bool caseInsensitive, CancellationToken ct);
    ValueTask<object?> CreateAsync(string list, string value, CancellationToken ct)   // for AutoCreate; optional
        => throw new NotSupportedException(...);
}

Two ship in the box: CodeListLookupProvider (key "codelist", backed by loadstone.CodeLists/Codes, supports auto-create) and SqlQueryLookupProvider — a configuration-defined provider where you give it a key, a SQL query taking @List/@Value, and optionally a separate connection string. That's how an existing company lookup database plugs in with zero code (see the "company X" section of the architecture doc).

CachingLookupResolver — policy & caching

Routes a field to its provider by key, caches hits in IMemoryCache, and applies the OnMissing policy to a miss. The provider only answers "what does this mean?"; everything else — cache keys, case handling, the four missing-value policies — lives here, so a new provider stays tiny.

11. The durable queue

Loadstone.SqlServer/SqlImportJobQueue.cs · SystemSchema.cs

The queue is a single SQL table, loadstone.Jobs. No broker, no extra infra — which is exactly why it works on App Service and on-prem out of the box. The job lifecycle is the most safety-critical code in the system, so understand it before touching it.

Claiming — atomic, multi-worker safe

Any number of workers, across processes and machines, poll the same table. The claim is a single statement that can never hand one job to two workers:

WITH next AS (
    SELECT TOP (1) * FROM loadstone.Jobs WITH (ROWLOCK, UPDLOCK, READPAST)
    WHERE QueueName = @Queue AND Status IN ('Pending','Failed')
      AND (NextAttemptAt IS NULL OR NextAttemptAt <= SYSDATETIMEOFFSET())
    ORDER BY CreatedAt)
UPDATE next SET Status='Processing', StartedAt=SYSDATETIMEOFFSET(),
    HeartbeatAt=SYSDATETIMEOFFSET(), Attempt=Attempt+1
OUTPUT inserted.*;

READPAST skips rows another worker has locked; UPDLOCK holds the claim; OUTPUT returns the claimed row. One indivisible operation.

Fencing — the subtle part

Do not remove the attempt fence CompleteAsync and FailAsync update WHERE Id=@Id AND Status='Processing' AND Attempt=@Attempt. This matters because a slow worker's job can be reclaimed (below) and re-run by another worker. Without the fence, the stale worker finishing late would clobber the new attempt's result. The fence makes its update a no-op (logged as a warning). Removing it reintroduces silent data races under retries.

Heartbeats & reclaim — telling slow apart from dead

A worker pulses HeartbeatAsync every 60 s while processing. ReclaimAbandonedAsync requeues jobs whose COALESCE(HeartbeatAt, StartedAt) is older than the abandoned-job timeout (default 30 min). Using the heartbeat, not StartedAt, is what lets a legitimately long import run for hours without being stolen and double-processed. On graceful shutdown a worker calls ReleaseAsync, which hands the job straight back to Pending without consuming an attempt — so deploys don't strand jobs for the reclaim window.

12. Queue workers

Loadstone.Runtime/Workers/QueueWorkerService.cs · LoadstoneInitializer.cs

QueueWorkerService is a BackgroundService that hosts one polling loop per queue slot. It periodically scans the dataset registry and starts loops for any queue it isn't already serving (respecting each dataset's concurrency). Each loop claims a job, opens a fresh DI scope, resolves an ImportEngine, runs it, and records the terminal state.

The bug that was here — don't reintroduce it The loops are stored in a ConcurrentDictionary. An earlier version passed the running task into TryAdd as an argument, which starts it before the duplicate check — spawning a new infinite loop on every scan. Always check membership then start the loop. This is called out in a comment at the call site.

A separate ImportEngine is resolved per job from a scoped container, so nothing scoped is shared across threads. LoadstoneInitializer (an IHostedService) runs at startup: it ensures the system tables exist (retrying while the database warms up) and loads persisted manifests into the registry.

13. The SQL writer

Loadstone.SqlServer/SqlServerImportWriter.cs · DataRecordReader.cs

The writer is where correctness and performance meet. It never touches rows one at a time; it stages in bulk and applies set-based SQL. It picks one of three strategies from the manifest's shape, automatically — the caller never chooses.

Strategy selection

// Flat dataset (root has no children): stream straight into a staging table + one MERGE.
if (context.Manifest.Root.Children.Count == 0)
    return await WriteFlatStreamedAsync(...);

// Hierarchical + some entity has no natural key ⇒ its inserts aren't idempotent,
// so a retry could duplicate. Run the whole job in ONE transaction.
var anyEntityLacksNaturalKey = context.Manifest.EnumerateEntities().Any(e => e.NaturalKey.Count == 0);
if (anyEntityLacksNaturalKey)
    return await WriteJobScopedAsync(...);

// Every entity has a natural key ⇒ merges are idempotent, partial commits are safe.
// Cheaper transaction-per-batch.
return await WriteBatchScopedAsync(...);

The unifying principle: each strategy is the fastest one that is still safe to repeat for that shape, because the queue is at-least-once. Flat = one atomic statement; keyless hierarchy = one job transaction; keyed hierarchy = idempotent per-batch merges.

Hierarchical merge — how children get their foreign keys

For a batch, records are flattened per entity (parents first, via EnumerateEntities). For each entity the writer: creates a #stage temp table, bulk-copies the rows into it (with a _StageId and, for children, the parent's already-known _ParentKey), then runs a MERGE whose OUTPUT captures each row's database key against its stage id:

MERGE [dbo].[OrderLines] WITH (HOLDLOCK) AS t
USING #stage AS s ON t.[OrderId] = s._ParentKey AND t.[LineNumber] = s.[LineNumber]
WHEN MATCHED THEN UPDATE SET ...
WHEN NOT MATCHED THEN INSERT (...) VALUES (...)
OUTPUT $action, inserted.[LineId], s._StageId INTO #out (...);

The writer reads #out, sets each record's DatabaseKey, and those keys become the _ParentKey values staged for the next level down. That's the entire foreign-key mechanism — no ORM, no per-row round trips, works at any depth.

Performance detail worth knowing For required natural-key columns the ON clause uses plain t.c = s.c; only nullable keys get the null-safe (t.c = s.c OR both IS NULL) form, which defeats index seeks. This one change was a ~9–15× speedup on large keyed tables. Batches flush at 500 roots or 50,000 total records, so a deep hierarchy (thousands of records per root) stays memory-bounded.

Flat fast path — the accelerator

WriteFlatStreamedAsync handles single-entity datasets (the big commercial-file case). Records stream from the parser through DataRecordReader — a custom DbDataReader — directly into SqlBulkCopy, filling one staging table for the whole file with no intermediate buffering, then a single MERGE (or INSERT..SELECT for keyless data) applies it. Constant memory at any size, and the single atomic statement makes it perfectly retry-safe. Measured ~30,000 records/s on a laptop against a 1M-row, 527 MB file.

SQL identifiers Every table/column name that reaches SQL goes through SqlIdentifier.Quote (bracket wrap + ] doubling) or EntityDefinition.QualifiedTable (same escaping). If you emit new SQL that embeds a manifest-supplied name, use those helpers — never interpolate a raw name. Values are always parameterized.

14. Schema management

Loadstone.SqlServer/SqlServerSchemaManager.cs · SystemSchema.cs

ISchemaManager does two jobs. EnsureSystemObjectsAsync creates Loadstone's own tables (loadstone.Jobs, JobEvents, RejectedRows, CodeLists, Codes) idempotently — each statement in SystemSchema.Statements is guarded with IF OBJECT_ID … IS NULL or IF COL_LENGTH … IS NULL, so it doubles as a lightweight migration runner (that's how the HeartbeatAt column is added to existing databases). It also creates the target database itself if the login can, by catching SQL error 4060 and issuing CREATE DATABASE against master.

GenerateTargetSchemaScript derives CREATE TABLE DDL for a dataset from its manifest: identity key, parent FK, typed columns, and a unique index over the natural key (scoped by the parent key). Exposed at GET /api/datasets/{name}/schema and applied by ApplyTargetSchemaAsync. Pre-existing tables work too — the manifest just maps onto them and schema generation is optional.

16. Observability

Loadstone.Runtime/Diagnostics/LoadstoneDiagnostics.cs

One class owns the telemetry names. It exposes an ActivitySource and Meter both named "Loadstone", plus counters (records read / rejected, rows written) and a job-duration histogram. Wire them into any OpenTelemetry pipeline with AddSource("Loadstone") / AddMeter("Loadstone"); the host already does, and setting OTEL_EXPORTER_OTLP_ENDPOINT exports both signals to any collector. Every job carries a CorrelationId that flows into log scopes, the trace, and the job-events timeline, so a single import is traceable end to end. Structured job history lives in loadstone.JobEvents; row-level failures in loadstone.RejectedRows, both queryable over HTTP.

19. Testing

Loadstone.Tests/* · scripts/smoke-test.ps1

Two tiers. Unit tests (NUnit, ~80, no database) cover the manifest model and validation, the value converter, all three readers (including the CSV parser's edge cases), the pipeline steps and lookup policies, the engine end-to-end with fakes, and — by asserting on the generated SQL strings — the merge/DDL generation. Run with dotnet test; CI runs them with warnings-as-errors.

End-to-end: scripts/smoke-test.ps1 drives a running instance through all three formats, an idempotent re-import, and a bad-row file, asserting status, counts, and rejection detail. Point it at any instance: .\scripts\smoke-test.ps1 -BaseUrl http://localhost:8080. It needs a real database (LocalDB, a container, or Azure SQL) — the layer that inspection alone cannot verify.

SQL generation is tested by string assertion The generated MERGE/DDL is verified by asserting on the produced SQL text, not by executing it. That catches structural regressions cheaply but not engine-level behavior — always run the smoke test against a real database after changing the writer or schema manager.

20. Maintainer gotchas

The short list of things that will bite you if you don't know them:

  • Entity names are globally unique per dataset. A segment that appears under two different parents (as RDBES's BV does under both SA and FM) can't be modeled directly yet. Per-parent entity naming is a known roadmap item.
  • The queue is at-least-once. Any new write path must be safe to repeat. If you add one, decide its idempotency story explicitly (see the three writer strategies).
  • Don't loosen the attempt fence on CompleteAsync/ FailAsync, and don't reclaim on StartedAt instead of the heartbeat — both reintroduce double-processing.
  • Readers must stream. The one documented exception is wrapped-object JSON. Don't add more ToList()-over-the-file shortcuts.
  • Validate at registration, not at runtime. If a manifest mistake can produce broken SQL, add a rule to DatasetManifest.Validate() with a clear message rather than letting it fail against the database.
  • All SQL identifiers from manifests go through the quoting helpers; all values are parameterized. No exceptions.
  • The Core project stays dependency-free. If a change wants to add a package to Core, it almost certainly belongs in a provider instead.
